One of the reasons kubernetes is so popular among the BigTech companies is because of its ability to scale. It can handle 1M requests per second. It comes with built-in load balancing and replicaSets which ensure availability at all times.
BigTech Companies love microservices because of their availability. When you have an application as complicated as Facebook, Amazon, Google it is very likely that there will be small bugs potentially breaking the application. Microservice prevents this by isolating the application into smaller services. Kubernetes along with Docker makes it easier to develop microservices applications.

Copy Paste the following command into your terminal/cmd to setup the project and authentication service. I would recommend to go through the packages we are installing and folder structure to get a better understanding of the project.

Now that we have the project setup, we can start by creating skaffold config file. The config file we are about to write provides skaffold, the location of kubernetes config files along with files to sync while building the image. Everytime there's a change in the code, skaffold will redeploy the image with the new code.
Coming to the auth-depl.yaml file, it contains the configuration for kubernetes of the authentication service. There's nothing much to explain here, I would recommend to go through kubernetes official documentation on this if you have any doubts.

apiVersion: skaffold/v2alpha3
kind: Config
deploy:
  kubectl:
    manifests:
      - ./infra/k8s/*
build:
  local:
    push: false
  artifacts:
    - image: cygnetops/auth
      context: auth
      docker:
        dockerfile: Dockerfile
      sync:
        manual:
          - src: "src/**/*.ts"
            dest: .

Ingress Ngnix Setup

We now have the authentication service running but we have issue here. There's no actual way to reaching out to our service. We know our service is running on port 3000, but if you visit localhost:3000, you will get a site can't be reached error. This is because our service is running on a VM. We can't reach out to the service from the outside. The only way to get this job done is to use a routing service which exposes our application to the outside world. We're gonna use ngnix to do this.

  • Install Ingress Ngnix

kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.1.1/deploy/static/provider/cloud/deploy.yaml

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress-srv
  annotations:
    kubernetes.io/ingress.class: "nginx"
spec:
  rules:
    - host: myMicroservicesApp.com
      http:
        paths:
          - path: /api/users/?(.*)
            pathType: Prefix
            backend:
              service:
                name: auth-srv
                port:
                  number: 80

As you might have noticed in the ngnix setup, we have passed an option called host. This option allows us to specify the hostname of our application. We have to locally route this hostname to where our application is running. This is because we are using ngnix and not for any other reason.
In this case, we have to edit the file /etc/hosts for Mac and c:\Windows\System32\Drivers\etc\hosts for Windows.

127.0.0.1 myMicroservicesApp.com

Database and Persisent Volume Claims

The best part of kubernetes is that it provides total control over the resources it uses, particuarlly for databases. We don't need a database running on a cluster in US or UK anymore. We can deploy our local database with kubernetes that is available at all times to our services.
That means we don't need to waste our money on a database service that is running in a different part of the world.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: auth-mongo-depl
spec:
  replicas: 1
  selector:
    matchLabels:
      app: auth-mongo
  template:
    metadata:
      labels:
        app: auth-mongo
    spec:
      containers:
        - name: auth-mongo
          image: mongo
---
apiVersion: v1
kind: Service
metadata:
  name: auth-mongo-srv
spec:
  selector:
    app: auth-mongo
  ports:
    - name: db
      protocol: TCP
      port: 27017
      targetPort: 27017

If play around with this application you will find out that data in our database will be lost if the application is restarted. To solve this, we need to create a persistent volume claim. This is a volume that is persistent across restarts of the application.
